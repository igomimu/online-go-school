# Lessons — online-go-school

## 2026-06-13: E2E緑でも実ユーザーで死ぬ「状態乖離」パターン
**事象**: 先生PWサーバー再設定後、三村さんが正しい新PWでログイン不能。E2Eは13/13緑だった。

**根因**: ローカル localStorage のPW照合がサーバー認証より先に走るゲートが残っていた。
E2Eヘルパーは毎回 `setupTeacherPassword(TEST_TEACHER_PASSWORD)` でローカルとサーバーを
**同じ値に揃えて**いたため、「ローカル≠サーバー」という実ユーザーの状態を一度も踏まなかった。

**教訓**:
1. **二重に保存された真実（ローカルキャッシュ vs サーバー）があるなら、E2Eに必ず「乖離ケース」を入れる**。
   揃った状態だけのテストは、揃っていることのテストにしかならない。
2. サーバー側の認証情報を変更したら、「既存ブラウザ状態（古いキャッシュ）からの初回ログイン」を必ず実機 or E2Eで踏む。
3. 認可の権威を移行するときは、旧権威のコード（照合・リセットUI）を**残さず撤去**する。
   並存させると「どちらかが通れば入れる/どちらかが落ちれば弾く」の混在で必ず事故る。
   → 対応済み: `e2e/teacher-auth.spec.ts` に乖離ケース2本を常設（commit 481b962）。

## 2026-06-30: 競合分析でブランド資産を機能比較に矮小化しない
**事象**: online-go-school の市場比較で、横浜囲碁サロン・ダイヤモンド囲碁サロン・洪道場の強みを「商品設計」「講師陣」「本格派」程度に薄くまとめてしまった。

**根因**: 競合の選ばれる理由を、機能・価格・提供形態に寄せすぎた。実際には、横浜囲碁サロンは囲碁YouTubeでの成功と発信者への信頼、ダイヤモンド囲碁サロンは都心一等地の高級サロンと女性アマチュアインストラクターの魅力、洪道場は日本最強棋士を何名も排出した実績ブランドが中核。

**教訓**:
1. 教室・サロン・教育サービスの競合分析では、機能表の前に「なぜその人・場が選ばれているか」を書く。
2. ブランド、人物像、場所の空気、実績、継続接触は、価格や機能と同等以上の購買理由として扱う。
3. 三村側の比較では、システム利便性だけでなく、三村九段の人柄・教室の継続感・保護者の安心感を必ず評価軸に入れる。

## 2026-07-08 「スクショで確認済み」と言いながら画像を見ていなかった
- **事実**: PWAボタンの本番検証でinnerText検出とconsoleログだけを見て「両画面で確認済み」と報告。実際は①先生の実動線（ログイン直後の教室管理画面）にボタンが存在しない②下部バーが#rootのpadding分（64px）ビューポート外に見切れ、の2点を見逃した。三村さんの実機指摘で発覚。
- **教訓**: スクリーンショットを撮ったら必ず自分の目で画像を開いて見る。テキスト存在≠ユーザーに見えている。「ユーザーが実際に通る動線」（ログイン後にどの画面に着地するか）で確認する。見切れはgetBoundingClientRect vs innerHeightで機械検証できる。

## 2026-07-09 「手番でない側でも石を置ける」が何度も直らなかった
- **事実**: 対局で手番側だけが着手できる仕様が繰り返し直らなかった。真因は、先生が対局のプレイヤー（黒/白）を兼ねると `isTeacher` フラグが手番チェックを全て素通りさせ、かつ `effectivePlayer` が先生自身の色で着手していたこと（黒手番でも白石が置ける）。サーバー(submit_move)は step3 で手番を検証し409で拒否していたが、クライアントが楽観的更新+LiveKit配信を先に走らせるため、相手の盤面に石が残って食い違い、「置けてしまう」ように見えていた。
- **教訓**:
  1. 「バリデーションが効かない」系は、まず権威(サーバー)が本当に通しているのか確認する。ここではサーバーは正しく拒否していた＝真の症状は「楽観的更新/ブロードキャストのロールバック漏れによる盤面desync」。「サーバーが検証していない」と決めつけない。
  2. ロールのバイパス（isTeacher で全チェック免除）と、色の導出ロジック（effectivePlayer=myColor）が別々にあると、両者の組み合わせで抜け穴ができる。権限フラグを「全部免除」に使わず、条件を分ける（isProxyTeacher = isTeacher && !isParticipant）。
  3. 手番系は「打とうとしている色 === その局面の手番の色」という単一の不変条件をコードで明示すると、ロールに依存せず封じられる。

## 2026-07-10: mainの巻き戻しを「事故」と決めつけて復元した誤判断
- 別AIのcommitがmainから強制的に消えていたのを「force push事故」と推定し、cherry-pickで復元してしまった。実際は三村さんが指示した意図的なrevert（別AIがネット囲碁学園=igocampusの問題を三村囲碁オンラインと錯覚して実装したため）。
- **2026-07-11: 状態変数 (State) のセッター消し忘れと自動切替競合バグ**
  - **事象**: `activeGameSource` を削除した際に、`setActiveGameSource` の呼び出し箇所が残っていたため、実行時に生徒側画面がクラッシュし、E2Eテストがタイムアウトした。また、手動で「手番ではない盤」に切り替えた際、自動遷移 useEffect が即座に手番の盤へ引き戻してしまい、E2Eテストが落ちた。
  - **教訓**:
    1. 状態変数を削除した際は、対になるセッター関数の消し忘れがないか徹底的に grep で確認する。コンパイルが成功しても実行時エラーで画面クラッシュの原因となる。
    2. 自動切替 useEffect が手動アクションと競合する場合、対局状態ハッシュ（手番と手数）を監視し、「対局自体に進行があった時のみ自動遷移を判定する」設計にすることで、手動選択による切り替えを阻害せず競合を回避できる。

## 2026-07-14: git add -A で意図的な未追跡ファイルを誤コミット
- **事故**: `git add -A src e2e ...` が、意図的にgit管理外としていた検証道具（e2e/proof-*.spec.ts）を拾い、publicリポジトリにpushした。秘密情報は無かったが運用違反。
- **ルール**: commit前に `git status -s` の `??`（untracked）を必ず確認し、追跡対象は**ファイル名を明示して add** する。ディレクトリ単位の `git add -A <dir>` は untracked を巻き込む前提で使う。意図的な未追跡ファイルは発見した時点で .gitignore に載せる（今回 proof-*.spec.ts を追加済み）。
